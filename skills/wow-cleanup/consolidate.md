# 合并与归档逻辑

整理文档时遵循以下判断维度和执行策略。具体阈值、读取深度、对比方式由模型视情况自行决定。

## 判断维度

| 维度 | 描述 | 默认操作 |
|------|------|----------|
| 重复 | 同一类型目录下多个文档主题指向同一功能 / 模块 / 接口 | 合并 |
| 过时 | 文档引用的代码 / 接口已不存在或已大幅重写，或描述的行为与现有实现明显不符 | 归档（更新或并入） |
| 碎片 | 同一主题被拆散在多个内容稀薄的小文档中 | 合并 |
| 错误归类 | 文档实际内容与所在目录类型不匹配（spec/api/acceptance/tmp/reference/sql） | 移动 |
| 空文档 | 没有实质信息 | 删除 |
| 临时文档过期 | `docs/tmp/` 下显著久未更新的临时文档 | 删除（用户可选择移动到 `spec/` 保留） |

判断方法提示：

- **重复 / 碎片**：阅读相关文档后比较主题归属，必要时全文对比
- **过时**：从文档中提取代码路径、接口、模块名等具体引用，用 Glob/Grep 在项目代码中验证；引用大量缺失或语义明显落后时标记为过时
- **错误归类**：按文档实际内容重新判断所属类型，与当前目录对比
- **临时文档过期**：参考 git log 时间和项目活跃度判断

## 执行策略

### 合并

挑选内容最完整的文档作为基础，将其他文档中的独有信息补充进去，写入后删除被合并的文档。文件名按内容命名（遵循 CLAUDE.md 中的命名规范），原名仍合适则保留。

### 归档

针对过时文档，把仍有价值的内容整理回当前文档体系，二选一：

- **就地更新**：直接修改原文件以反映当前代码 / 接口的实际状态
- **并入其他文档**：将相关内容合入现有当前文档（不存在则新建合适的文档），删除原过时文档

判断方式：原文件主题在新代码中仍然成立 → 就地更新；主题已被其他文档覆盖或需要重新组织 → 并入。

### 移动

将文件移动到正确的类型目录。目标目录已有同名文件时：主题相同则更新已有文档，不同则加限定词区分。

### 删除

直接删除文件。
