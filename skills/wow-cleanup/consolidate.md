# 合并与归档逻辑

## 判断规则

### 重复文档检测

同一类型目录下，两个文档满足以下任一条件视为重复：
- 标题中的主题关键词重叠度 > 70%
- 正文前 20 行涉及相同的模块/功能/接口

**建议操作：合并** — 保留内容更完整的文档作为基础，将另一个文档中的独有信息补充进去，删除被合并的文档。

### 过时文档检测

文档中引用的文件路径或接口，用 Glob/Grep 在项目代码中搜索不到。

**判断流程：**
1. 提取文档中出现的文件路径（如 `src/auth/middleware.ts`）和接口路径（如 `POST /api/user/list`）
2. 在项目代码中搜索这些路径
3. 如果 > 50% 的引用已不存在，标记为过时

**建议操作：归档** — 移动到 `docs/_archive/`，保留文件但不再参与日常查阅。

### 碎片文档检测

同一主题的信息分散在 3 个或以上小文档中（每个不足 30 行）。

**建议操作：合并** — 整合为一个完整文档。

### 错误归类检测

文档内容的主要类型与所在目录不匹配。例如：
- `spec/` 下的文档实际内容是独立 API 参考 → 应在 `api/`
- `spec/` 下的文档实际内容是验收断言和执行结果 → 应在 `acceptance/`
- `api/` 下的文档实际内容是需求分析和任务计划 → 应在 `spec/`

**判断方法：** 按 [SKILL.md](../wow-doc/SKILL.md) 中"分类判断要点"重新判断文档类型，与当前目录对比。

**建议操作：移动** — 移动到正确的类型目录。

### 空文档检测

文档内容不足 5 行（排除空行）。

**建议操作：删除** — 直接删除。

### tmp 过期检测

`docs/tmp/` 下的文档，通过 `git log -1 --format=%ci <file>` 检查最后修改时间，超过 30 天未更新的标记为过期。

**建议操作：删除** — 临时文档过期后直接删除。用户也可选择移动到 `spec/` 保留。

---

## 执行策略

### 合并

1. 读取所有待合并文档的全文
2. 选择内容最完整的文档作为基础
3. 从其他文档中提取基础文档没有的信息，补充到对应章节
4. 按 [naming.md](../wow-doc/naming.md) 规则为合并后的文档命名（如原名仍合适则保留）
5. 写入合并后的文档，删除被合并的文档

### 归档

1. 确保 `docs/_archive/` 目录存在
2. 将文件移动到 `docs/_archive/`（保持原文件名）
3. 同名冲突时在文件名后追加日期：`xxx-20260416.md`

### 移动

1. 将文件从当前目录移动到正确的类型目录
2. 检查目标目录是否有同名文件，有则按 [naming.md](../wow-doc/naming.md) 冲突规则处理

### 删除

直接删除文件。用于空文档和过期的 tmp 文档。
