# /wow-doc — 按需文档管理

按类型自动归类文档，AI 根据内容命名，不限制文档内容和结构。

```
触发 → 环境检查 → 意图识别 & 分类 → 生成/更新文档
```

详细规则：
- 命名规则与冲突处理：[naming.md](naming.md)
- 异常处理：[error-handling.md](error-handling.md)

---

## 文档类型体系

按类型分目录存储在目标项目的 `docs/` 下：

| 目录 | 中文名 | 用途 |
|------|--------|------|
| `spec/` | 需求文档 | 需求分析、实施计划、技术方案等开发文档 |
| `api/` | 接口文档 | 独立的 API 参考文档 |
| `acceptance/` | 验收文档 | 验收计划和验收结果 |
| `tmp/` | 临时文档 | 草稿、快速笔记、一次性内容 |

**分类判断要点：**
- 大多数开发文档归入 `spec/`，这是默认类型
- 只有独立的 API 参考文档才归入 `api/`（如果 API 描述是 spec 文档的一部分，不需要拆出来）
- 定义验收标准、记录验收结果 → `acceptance/`
- 用户明确说"临时记一下"、"草稿"、"先随便写"等，或内容明显是一次性的 → `tmp/`

---

## Step 0 — 环境检查

检查项目根目录下 `docs/` 是否存在，不存在则创建。

类型子目录（如 `docs/spec/`）在 Step 1.2 确定文档类型后再按需创建。

---

## Step 1 — 意图识别与分类

### 1.1 确认输入

检查用户是否提供了具体描述。

**无输入时**，提示并等待：

```
请描述你的需求，例如：
  · 给用户列表增加按作者名搜索功能
  · 记录支付模块的数据模型和接口契约
  · 写一份订单服务的 API 参考文档
  · 临时记一下刚才讨论的技术选型结论
```

### 1.2 判断文档类型

用户显式指定了类型（如"写一份 api 文档"、"记到 tmp"）→ 直接采用，跳过自动分类。

否则根据用户描述和上下文，按上方"分类判断要点"自动判断类型。

### 1.3 扫描已有文档

使用 Glob 扫描 `docs/<type>/` 下已有文件，判断：
- 是否有内容高度相关的已有文档 → **更新**
- 无匹配 → **新建**

扫描时读取每个文件的标题（`# ` 开头行）和前 20 行内容作为摘要，与当前需求做语义比对。

如果存在多个可能相关的文档，向用户展示选择：

先调用 `ToolSearch`（query: `"select:AskUserQuestion"`）获取工具 schema，然后调用 `AskUserQuestion`：
- `question`: `"发现以下相关文档，请选择操作"`
- `header`: `"文档匹配"`
- `options`: 列出匹配文档 + "新建文档" 选项
- `multiSelect`: `false`

### 1.4 生成文件名

读取 [naming.md](naming.md) 中的命名规则，为新文档生成文件名。更新已有文档时跳过此步。

---

## Step 2 — 生成/更新文档

**新建文档：** 写入 `docs/<type>/<filename>.md`，内容和结构由 AI 自由组织。

**更新文档：** 读取已有文档全文，原地修改使其反映当前最新状态。不追加变更历史——git 跟踪历史。

**写入后输出确认：**

```
✅ 文档已写入：docs/<type>/<filename>.md
   类型：<中文类型名>
   操作：新建 / 更新
```
